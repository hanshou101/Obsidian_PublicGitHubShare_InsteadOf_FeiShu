# 1、调研结果


- 官方自我定位/诙谐介绍：
	- 【在深度生成世界中建造堡垒、让矮人生存、保持快乐、扩张社区并避免挖得太深】的殖民地/模拟/策略游戏。

- 1
	- 它的核心不是单纯【建造地下基地】，而是一个先经过长期历史模拟的奇幻世界：世界生成会创建地理、历史、语言、文明兴衰、战争、人物与个体性格，并且这些活动会在玩家游玩时继续影响世界。

- 1
	- 【玩法系统与工程实现规格】的口径：
		- 世界生成、堡垒模式、冒险模式、传说模式、竞技场、地图、材料、物品、工坊、物流、工作订单、心理、贵族、正义、医疗、军事、战斗、流体、温度、行业链、事件史书
		- Web 架构拆分。
- 1
	- 三大主模式：    （初期，先只做【堡垒模式】；集中精力，把最重要的模式做好。）
		- 堡垒模式本身是三大主模式中最常被默认指代的模式；
			- 玩家选择 embark 地点、配置最初七名矮人的技能与物资，然后通过设计和命令间接管理殖民地，而不是逐个直接控制矮人。 
		- Adventure Mode 冒险模式，则让玩家在生成世界中创建角色或队伍，探索城镇、堡垒、荒野、洞穴、巢穴、遗迹，并可重访旧堡垒；
			- 当前版本下 Adventure 中建造不可用，制作能力也较有限。 
		- Legends Mode 是已生成世界的史书浏览器，可查看地图、文明历史和任何生卒人物，玩家在 Fortress/Adventure 中的重要成就也会被记录。
		- Object Testing Arena 则适合做开发调试入口：可生成生物、分配装备/技能、设置队伍、生成水/岩浆/树、调整天气和温度。

## 推荐的系统分层

可以把开发对象拆成 13 个一级域：
- **World Generation**：种子、区域地图、气候、地形、地质、文明、历史人物、站点、神器、事件。
- **3D Tile Map**：Z-level、tile、地形、构造物、建筑、物品、生物、流体、温度、可通行性。
- **Job / Labor / Skill Engine**：designation、工坊任务、work orders、zone jobs、need jobs、劳动许可、技能成长、资源预定。
- **Logistics**：stockpile、搬运、容器、take/give links、forbid/dump/melt/trade 标记。
- **Industry**：木、石、农业、酿酒、厨房、动物、皮革、纺织、金属、玻璃、陶瓷、肥皂。
- **Creature & Psychology**：种族、身体结构、属性、技能、人格、价值观、偏好、需求、想法、记忆、压力。
- **Rooms / Zones / Locations**：卧室、餐厅、办公室、墓室、会议区、酒馆、神庙、图书馆、公会、医院。
- **Administration**：manager、bookkeeper、broker、mayor、nobles、mandates、demands、justice、messenger。
- **Military & Combat**：squad、uniform、schedule、barracks、训练、近战、远程、护甲、伤口、围攻。
- **Healthcare**：hospital、诊断、清洗、手术、缝合、接骨、包扎、牵引、夹板、拐杖、肥皂。
- **Fluids / Temperature / Weather**：水深 0–7、压力、泵、冻结、岩浆、黑曜石反应、火、烟、蒸汽。
- **Story / Legends / Events**：历史事件日志、公告、神器、strange mood、雕刻、书籍、Legends browser。
- **Adventure Mode / Arena**：角色创建、小队、旅行、传闻、任务、战斗、测试生物/装备/天气/流体。


## 最重要的工程判断

第一阶段不要追求【全量 1:1】。最危险的四个复杂度黑洞是：**物流路径、流体压力、战斗组织层、心理记忆系统**。

建议先实现一个稳定的最小闭环：
- 64×64×16 或类似规模的局部 3D tile map。
- 7 个矮人、挖掘、砍树、建造、工坊、stockpile、搬运。
- 简化食物/酒/睡眠需求。
- 农场、酿酒、床、桌椅、餐厅、卧室。
- Job reservation 与路径查找。
- 基础 thought/stress。
- 一次商队、一次移民、一次小型敌袭。
- 所有重大事件写入 Legends-like timeline。

## 进阶和需要注意的重要内容

- Work orders 应尽早做，因为它是堡垒自动化生产的中枢；
	- Wiki 将其描述为 workshop management 的高级功能，可由 Manager 解锁，用来自动化或精细控制堡垒生产活动。
- Labor 与 Skill 要分开建模：
	- job 是具体任务，labor 决定能否做，skill 表示熟练度，执行任务会提升技能。

### 需要重点数据化的规则

- 材料系统必须做成数据驱动。
	- 密度会与体积共同影响物品重量，进而影响搬运与战斗；材料定义还涉及温度、熔点、燃点、magma-safe/fire-safe、冲击/剪切强度等。 
	- 战斗中，武器接触面积、穿透面积、材料和组织层共同影响结果；低接触面积更利于穿甲，高接触面积影响更多组织。
- 心理系统也必须数据化。
	- 矮人的正面 thought 会降低压力，负面 thought 会提高压力；needs 影响 focus，长期未满足可能转化为坏想法。 
	- Stress 页面进一步说明，矮人会维护一个压力分数，情绪体验与长期记忆都会改变它。
- 农业和酒是生存核心：
	- 食物来源可包括农业、贸易、捕鱼、狩猎、采集、畜牧繁殖和陷阱；稳定食物通常以农业为主，而酒/可酿植物也高度依赖农业、贸易或采集。
	- 金属工业则应分阶段实现：
		- 矿石先经 smelter 变成金属条，许多金属加工需要 fuel；steel 需要 iron、flux stone 与 fuel。
- 医疗系统中：
	- 医院 location 会征用 thread、cloth、splints、crutches、plaster powder、buckets、soap 等物资；伤员治疗包括休息、诊断、清洗、手术、缝合、接骨、包扎、牵引和固定等步骤。 
- 流体系统可后置，但架构要预留：
	- Dwarf Fortress 的水有压力机制，流体可向上运动直到压力平衡；岩浆与水类似但有特殊温度和黑曜石反应。

## 3.2、最重要的整合结论

**攻略最有价值的地方，不是“教玩家点哪个按钮”，而是揭示这个游戏的真实流程骨架：**

> 世界生成 → 安全选址 → 七矮人和物资 → 挖入口 → 仓库 → 食物/酒 → 工坊 → 房间 → 管理者/账本员 → work orders → 贸易 → 防御 → 金属工业 → 医疗 → 社会空间 → 事故/失败/Legends 复盘。

- PeridexisErrant 的 walkthrough 很适合给 AI 建立这种“章节式学习顺序”，
	- 因为它覆盖了每座堡垒都会需要的核心主题，并把更复杂的 beekeeping、soap-making、digging too deep、modding 等作为后续扩展。

**评论和故事最有价值的地方，是告诉我们 Web 版应该特别重视“可检查性”和“因果链”。** 

- 很多玩家故事并不是游戏直接写出来的，而是玩家通过 combat logs、thoughts、personalities、relationships、rumors、Legends Mode 等信息自己拼出来的。

因此，我（GptPro的AI）在文件里特别建议 Web 版做三个系统：

1. **Why Not? Advisor**：解释任务为什么没执行。
2. **Story Trace EventLog**：记录 actor、target、item、location、material、cause_event。
3. **Legends-lite / Postmortem**：让失败、事故、英雄行为、洪水、医疗失误、酒馆斗殴都能被事后追踪和分享。

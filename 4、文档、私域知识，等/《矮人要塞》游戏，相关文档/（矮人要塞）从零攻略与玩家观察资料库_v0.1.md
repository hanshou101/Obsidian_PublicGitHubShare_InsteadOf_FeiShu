# 《矮人要塞》从零攻略与玩家观察资料库 v0.1

创建日期：2026-07-01  
用途：为“Web 端网页版《矮人要塞》类游戏”的 AI Coding / Vibe Coding 工具提供更具体的玩法流程、玩家痛点、攻略路径和涌现叙事素材。  
口径：本文件不搬运攻略原文，而是把资料抽象成“可开发的系统需求、教程步骤、UI 解释、测试场景和 source cards”。

---

## 0. 一句话结论

想让 AI Coding 工具真正理解《矮人要塞》，只读百科式机制清单还不够；必须让它理解“一个新手从零到第一座可持续堡垒”的行动顺序，以及玩家为什么会把失败、事故和荒唐细节讲成故事。

本资料库建议把资料分成四层：

1. **零基础攻略层**：Quickstart、PeridexisErrant、Steam Intermediate、PC Gamer/Explorminate 等，用来抽“玩家先做什么、后做什么”。
2. **系统专题层**：Work Orders、Stockpile Design、DFHack Quickfort/Dreamfort，用来抽“自动化、物流和蓝图”的工程模型。
3. **中文语境层**：灰机 Wiki、PeridexisErrant 中文翻译、Gcores 等，用来统一中文术语和感性描述。
4. **玩家故事层**：Steam 评论分析、Boatmurdered、醉猫、Stupid Dwarf Tricks、Megaprojects，用来抽“为什么这些系统会产生可讲述故事”。

---

## 1. 推荐阅读顺序

### 1.1 给策划 / 产品 / AI Coding 工具的最小阅读路径

**第 1 组：学会第一座堡垒**

- G01 Dwarf Fortress Wiki Quickstart Guide

**第 3 组：理解“为什么好玩”**

- G20 Gcores 中文新手指南

- G24 Boatmurdered
- G25 Stupid dwarf trick

- 使用备注: Useful because players explicitly explain how they inspect the game to find stories.

## 3. Canonical Zero-to-Fortress Flow：从零到第一座可持续堡垒

这部分是给 AI Coding 工具最重要的流程抽象。每一步都同时包含：玩家目标、应建模机制、Web/UI 要求、常见新手坑和资料来源。

### F00 — 学习心态：把失败当成学习回路

**玩家目标**：接受没有标准胜利条件，第一座堡垒的目标只是建立一个可持续小社区。

**应建模机制**：failure_persistence, announcement_log, post_mortem_summary

**Web/UI 要求**：
- 展示可恢复的失败摘要
- 提示玩家每次失败会留下世界历史

**常见新手坑**：
- 把首次堡垒当成必须完美通关
- 过早追求复杂机器/岩浆/军事

**资料依据**：G01, G03, G09, G21
### F01 — 创建世界

**玩家目标**：选择世界大小、历史年数、文明/站点/野兽/矿物/自然野性等参数。

**应建模机制**：world_seed, history_length, civilizations, sites, beasts, savagery, minerals

**Web/UI 要求**：
- 提供新手预设
- 显示性能成本
- 显示危险度/资源丰富度评分

**常见新手坑**：
- 新手选过大世界导致生成/运行慢
- 忽略自然野性和邪恶生物群系

**资料依据**：G01, G02, G08
### F02 — 选择启程地点 Embark

**玩家目标**：找一个安全、资源足够、可持续的地点，而不是一开始就挑战地狱难度。

**应建模机制**：biome, aquifer, trees, surface_water, soil, clay, sand, flux, iron, neighbors

**Web/UI 要求**：
- Embark Advisor：标红重/多层含水层、邪恶/凶恶区域、敌对邻居
- 推荐河流/树林/土层/非邪恶/非凶恶/可贸易文明

**常见新手坑**：
- 不知道含水层会阻碍挖掘
- 缺木头或缺土层导致早期农业/床/桶困难
- 与哥布林/塔太近

**资料依据**：G01, G08, G13
### F03 — 准备七名矮人和起始物资

**玩家目标**：带足镐、酒、食物、种子、斧/医疗/贸易关键物资，并设置基础技能。

**应建模机制**：starting_dwarves, skills, embark_items, tools, seeds, animals

**Web/UI 要求**：
- 显示物资用途和缺失警告
- 推荐至少两名矿工与足够工具
- 提供新手装载模板

**常见新手坑**：
- 忘记镐导致无法挖掘
- 忘记斧头导致无法砍树
- 过早带大量无用奢侈品

**资料依据**：G01, G03, G06
### F04 — 抵达后建立安全聚集点和初始 stockpile

**玩家目标**：让矮人和物资尽快从野外进入相对安全的位置。

**应建模机制**：meeting_area, stockpile, hauling_jobs, wagon_items, forbidden_items

**Web/UI 要求**：
- 一键创建临时室内 All stockpile 和 meeting area
- 高亮未搬运物资
- 显示搬运队列

**常见新手坑**：
- 以为能直接控制矮人
- 物资留在地表被动物/敌人/天气影响
- 把 refuse/corpses 放进主仓导致腐烂/瘴气

**资料依据**：G01, G06, G16
### F05 — 挖入口、楼梯、基础大厅

**玩家目标**：用 designation 间接下达挖掘命令，建立可扩展的垂直要塞骨架。

**应建模机制**：dig_designation, stairs, channeling, ramps, walkability, path_cost

**Web/UI 要求**：
- 明确区分挖掘/通道/楼梯
- 提供水体/含水层警告
- 显示通行图

**常见新手坑**：
- 把 channeling 当普通挖掘
- 挖穿水体或让水对角流入

**资料依据**：G01, G07
### F06 — 建立食物与酒的最小闭环

**Web/UI 要求**：

- 提供重复酿酒/做饭订单

**常见新手坑**：
- 只关注食物忘记酒
- 把种子或酒拿去烹饪

**资料依据**：G01, G06, G07, G15
### F07 — 建立第一批工坊和家具链

**玩家目标**：用 carpenter/stoneworker/mechanic 等工坊生产床、门、桌椅、箱、机制零件。

**应建模机制**：workshop, job_queue, item_quality, material_selection, input_stockpile, output_stockpile

**Web/UI 要求**：

- 显示缺材料/缺工人/缺技能原因
- 提供批量家具订单

**常见新手坑**：
- 工坊离材料太远导致搬运时间占满

**玩家目标**：建立 food/drink/seeds/wood/stone/bar blocks/refuse/corpses 等专用仓库。

**应建模机制**：stockpile_filters, give_to_take_from_links, bins_barrels, wheelbarrows, miasma

**Web/UI 要求**：

**常见新手坑**：

## 4. 从攻略抽出的“开发者视角”系统模型

### 4.1 Tutorial Step Model

建议每个新手引导步骤使用如下结构：

```ts
type TutorialStep = {
  id: string;
  title: string;
  playerGoal: string;
  prerequisites: GamePredicate[];
  successDetector: GamePredicate[];
  failureHints: FailureHint[];
  relevantSystems: SystemTag[];
  optional: boolean;
  sourceIds: string[];
};
```

这可以避免教程变成纯文本。举例：  
“建一个 still 并酿酒”这一步不只是提示文本，而是需要检测 `workshop.type === 'still'`、是否有可酿植物、是否有空桶/罐、是否有可执行 brewer labor 的居民、是否生成了 `BrewDrinkFromPlant` job，以及失败时能告诉玩家缺什么。

### 4.2 Job / Labor / Stockpile / Work Order 的四层关系

```text
Player intent
  ├─ designation: dig/build/zone/stockpile
  ├─ workshop job: make bed / brew drink / smelt ore
  ├─ work order: keep 20 beds / brew if drinks < 100
  └─ military/order: train / patrol / defend burrow

Simulation
  ├─ job reservation: reserve item, location, workshop, worker
  ├─ labor eligibility: dwarf can do this labor?
  ├─ pathfinding: can worker reach target and materials?
  ├─ material availability: is item permitted / forbidden / reachable?
  └─ execution: move → fetch → work → produce → store → log
```

Web 版最需要提前实现的是“为什么没执行”的解释层。新手最大的挫败不是复杂，而是：玩家看不到某个任务为什么卡住。

### 4.3 Stockpile 是经济系统的接口，不只是仓库

Stockpile 至少需要有：

```ts
type Stockpile = {
  id: string;
  tiles: TileCoord[];
  accepts: ItemFilter;
  giveTo: StockpileId[];
  takeFrom: StockpileId[];
  maxBins?: number;
  maxBarrels?: number;
  maxWheelbarrows?: number;
  priority?: number;
  allowPreparedFood?: boolean;
  allowSeeds?: boolean;
  allowRefuse?: boolean;
  allowCorpses?: boolean;
};
```

从攻略看，很多新手问题都来自 stockpile 规则：refuse 进入主仓会腐烂/瘴气，石头和木头占满空间，工坊输入太远导致搬运吞掉全部时间，种子和酒的许可设置会影响农业和酿酒循环。

### 4.4 Work Order 是中期自动化的核心

一个简化版也至少应支持：

```ts
type WorkOrder = {
  id: string;
  product: ItemSpec;
  quantity: number;
  repeat: "once" | "daily" | "monthly" | "seasonally" | "yearly";
  conditions: WorkOrderCondition[];
  workshopType?: WorkshopType;
  managerApprovalRequired?: boolean;
  status: "active" | "waiting" | "suspended" | "complete";
  generatedJobIds: string[];
};
```

最小 MVP 条件：

- `available_item_count(product) < threshold`
- `available_material_count(input) >= threshold`
- `empty_container_count >= threshold`
- `population >= threshold`
- `season in allowedSeasons`

## 5. 从评论/故事抽出的设计原则

### D01 — 攻略不是线性答案，而是把玩家带入自给自足循环

- 观察：最有价值的新手攻略都围绕“从车队到可持续堡垒”展开：地点、物资、挖掘、仓库、食物、酒、床、餐厅、工坊、贸易、防御。
- 开发转译：把 tutorial 做成一条可跳过的任务链；每个任务同时解释玩家意图、系统条件和失败原因。
- 来源：G01, G03, G06
### D02 — 间接控制是核心手感

- 观察：玩家通常不直接操控单个矮人，而是画区域、指定任务、配置规则；乐趣来自系统自动执行和出错后的调查。
- 开发转译：Job 系统必须包含 job reservation、labor eligibility、pathfinding、resource availability、cancellation reason，并在 UI 里可追踪。
- 来源：G01, G07, G21
### D03 — 物流和距离是隐藏主玩法

- 观察：很多攻略反复强调工坊、仓库、输入输出距离、wheelbarrows、垂直布局；这说明“少走路”本身就是策略。
- 开发转译：引擎中每个 job 都应有移动成本；UI 提供物流热力图和 stockpile-link 可视化。
- 来源：G06, G16, G17, G18
### D04 — Work Orders 是从新手手动操作到堡垒自动化的分水岭

- 观察：手动工坊队列适合教学，条件化 work orders 适合稳定中期运营。
- 开发转译：早期就实现一个简化版条件订单系统：目标物品、数量、周期、条件、失败原因、生成 job 列表。
- 来源：G06, G15, G18
### D05 — 玩家故事需要可读的因果链

- 观察：评论中的精彩故事往往来自系统间交互；玩家需要能从尸体、伤口、日志、思想、关系、地点历史中还原事件。
- 开发转译：每个重要事件都写入 EventLog，并保留 cause/ref links：actor、target、item、location、material、previous_event。
- 来源：G20, G21, G27, G28, G30
### D06 — 失败不是惩罚，是内容生成器

- 观察：水淹、含水层、怪物、斗殴、贵族要求、医疗失败、敌袭、物流崩溃都可以变成战报。
- 开发转译：实现 Reclaim/Retire/Legends-lite：堡垒失败后仍保留废墟、幸存者、神器、墓碑和事故时间线。
- 来源：G21, G23, G24
### D07 — 复杂系统只有在可检查时才有魅力

- 观察：游戏魅力来自“几乎每个 tile、物品、生物、关系、念头都能检查”，但新手也常被信息埋没。
- 开发转译：实现分层 inspector：默认展示三行结论，展开后展示完整状态、历史、规则来源和可行动建议。
- 来源：G20, G21, G30
### D08 — “愚蠢的矮人戏法”和巨构是后期目标替代品

- 观察：由于没有硬性胜利，玩家会自创金字塔、水泵、岩浆装置、巨型宫殿、Moria、机器等目标。
- 开发转译：不要只做生存；提供 blueprint、成就、材料统计、工程验证和性能安全阈值支持自定义目标。
- 来源：G20, G25, G26, G18

## 6. 有趣发现 / 经典故事 / 可做成测试场景的素材

### E01 — 醉猫 / 酒精清洁链

- 摘要：猫踩过溢出的酒，清洁身体时摄入酒精；体型/剂量/综合征规则共同造成醉猫甚至死亡的经典故事。
- 牵涉系统：liquid_spill, body_covering, self_cleaning_interaction, ingestion, syndrome_effect, body_size
- Coding take-away：涌现故事来自小规则组合；每一步都应可在 debug/story trace 中查看。
- 来源：G27, G28
### E02 — Boatmurdered

- 摘要：早期著名接力战报，包含大象、洪水、岩浆、商队事故、精神崩溃等多重失败链。
- 牵涉系统：wildlife, siege, flooding, magma, caravan, mood_or_stress, succession_narrative
- Coding take-away：不要写固定剧情；要让失败链、环境危险和玩家工程事故能自然叠加。
- 来源：G24
### E03 — 金字塔、水泵与血水表演

- 摘要：玩家用采矿、冶炼、建筑、泵、水/血流动和神殿设定创造高度个人化的工程艺术。
- 牵涉系统：construction, material_value, fluid, pump, location_assignment, ritualized_player_goal
- Coding take-away：系统应允许玩家把实用机制拼成非实用表达；不要把建筑限制成固定房间模板。
- 来源：G20, G25
### E04 — 医院与肥皂链

- 摘要：中期医疗不是单一按钮，而是一组床、诊断、清洗、手术、缝合、包扎、接骨、牵引、肥皂、桶、水源、布料等资源链。
- 牵涉系统：wounds, hospital_zone, healthcare_jobs, soap_industry, textiles, water
- Coding take-away：复杂系统应做成可分解 pipeline；UI 必须显示缺哪一步。
- 来源：G06
### E05 — Quickfort / Dreamfort

- 摘要：社区通过蓝图工具把完整要塞设计、仓库、房间、工坊、work orders 等结构化复用。
- 牵涉系统：blueprint_dsl, designation_batch, layout_preview, manager_orders, stockpile_rules
- Coding take-away：Web 版应把蓝图 DSL 做成一等公民：导入、预览、旋转、差异检查、撤销、版本化。
- 来源：G17, G18

## 7. 面向 Web 版的具体设计建议

### 7.1 把“新手攻略”做成游戏内 Advisor，而不是外部文档

建议实现三个 Advisor：

1. **Embark Advisor**：告诉玩家这个地点危险在哪里、资源好在哪里。
2. **Fortress Health Advisor**：显示食物、酒、床、房间、仓库、医疗、防御、幸福度的缺口。
3. **Why Not? Advisor**：当一个 job 不执行，直接显示原因：没有材料、材料被禁止、路径不可达、没有工人、工人忙、工坊被占用、库存判断不准确等。

### 7.2 把“玩家故事”做成可查询事件链

不要只存：

```json
{ "event": "Urist died" }
```

而要存：

```json
{
  "event": "creature_death",
  "actor": "goblin_12",
  "target": "dwarf_7",
  "location": "tile_42_18_3",
  "cause": "axe_strike",
  "weapon": "iron_battle_axe_2",
  "wound": "left_lung_punctured",
  "previous_events": ["ambush_55", "burrow_alert_3_failed_path"]
}
```

这样 Web UI 可以回答：

- 谁死了？
- 为什么死？
- 死前在做什么？
- 伤口来自什么武器？
- 是否由于玩家没开 burrow / 门没关 / 医院缺物资？
- 这件事是否应该进入 Legends-lite？

### 7.3 把 Quickfort 思路内建成 Web 原生能力

DFHack Quickfort 对 Web 版特别有启发：Web 游戏天然适合做拖拽、预览、模板、批量应用、撤销和分享。建议第一版 blueprint 支持：

- 多 Z-level。
- dig / build / place / zone / stockpile / burrow / order。
- 预览与冲突检查。
- 旋转、镜像、平移。
- 资源需求估算。
- 生成教程地图。
- 导出 JSON 供 AI Coding 工具读写。

### 7.4 让“荒诞”来自系统组合，而不是写死段子

不要硬编码“醉猫故事”本身；编码的是：

- 液体可以留在 tile 上。
- 生物身体部位可以被沾湿/污染。
- 生物有清洁行为。
- 清洁可能摄入附着物。
- 摄入物有剂量和身体尺寸修正。
- 生理效果可以改变状态。
- 事件链可被日志捕捉。

这样才会产生新的“醉猫式”故事，而不只是复刻旧段子。

## 10. 资料使用注意事项

1. **攻略版本会漂移**：Dwarf Fortress 多年持续更新，老攻略对概念非常有用，但 UI 和少数机制可能不再准确。
2. **中文资料适合统一语感**：中文社区资料能帮助团队理解，但规则实现仍应以当前 wiki/官方资料交叉核对。
3. **评论和故事不是机制规范**：Boatmurdered、醉猫、Stupid Dwarf Tricks 应作为“设计目标”和“测试灵感”，不是当前版本精确规则。
4. **AI Coding 不应直接复述攻略文本**：建议把攻略转成数据结构、流程图、UI 状态机和测试场景。
5. **优先实现可解释性**：矮人要塞类游戏最容易失败在“系统很复杂但玩家不知道为什么”。任何复杂系统都应有 inspector、event trace、cancellation reason 和 postmortem。

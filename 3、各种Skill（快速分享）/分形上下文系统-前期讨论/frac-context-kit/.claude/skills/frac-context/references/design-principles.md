# Frac Design Principles

## 设计目标

```text
低维护成本
低认知成本
低 token 噪声
可解释
可重建
可以被普通文件系统理解
```

## 系统边界

Frac Context 不是上下文平台。它只是一组文件约定和一个小工具。

它不尝试自动知道“最相关 token”。它只沿目录树维护递归摘要。

## 为什么不用数据库

因为当前系统唯一需要的状态是：

```text
.frac.md 是否晚于它的输入
```

文件系统 mtime 已经能表达这个状态。

## 为什么不用 hash

hash 可以更精确判断内容变化，但会引入缓存、数据库或状态文件。

本系统优先选择简单、透明、可手工修复的机制。

## 为什么不用 embedding / RAG

embedding / RAG 适合语义检索，但会引入：

```text
索引构建成本
召回漂移
向量库维护
chunk 策略
重建策略
权限和隐私问题
```

而 Frac Context 的目标是让目录树本身成为上下文结构。

## 为什么父级不读孙目录

一旦父级可以跨层读取，系统就会退化成全局总结器。

严格限制父级只读直属输入，才能保证：

```text
局部更新
局部责任
局部可验证
链式传播
```

## 判断系统是否走偏

如果出现以下现象，说明系统开始膨胀：

```text
根 .frac.md 超过 300 行
每次修改都触发大面积无意义重写
Agent 更新父级时经常读孙目录
需要人工频繁思考该给模型哪些 token
.frac.md 出现大量空话
引入数据库、embedding、图谱、复杂 schema
```

### 1. 背景与核心痛点

本质上是在解决「VibeCoding 项目的上下文膨胀悖论」：

- **规模无限 vs 窗口有限**：项目会持续生长，代码、设计决策、风格约定、隐性共识、历史踩坑记录会越堆越多；但不同模型的上下文窗口从几千到上百万 token 不等，没有一套能自适应的供给方案。
- **局部细节 vs 全局氛围**：只给模型看局部代码，它会丢失项目整体的 Vibe（命名风格、架构哲学、设计取舍），写出 “对但违和” 的代码；全量塞进去又塞不下，而且冗余信息会稀释核心意图。
- **静态摘要 vs 动态演进**：传统的 README、架构文档是人工写的静态快照，项目一变就过时；普通的 AI 代码摘要只总结功能，不沉淀 “为什么这么做” 的决策上下文，更保不住 Vibe。
- **RAG 的结构性缺陷**：现有检索增强是 “碎片化拼接”，搜出来的是零散片段，没有全局结构感，模型拿到的是一堆碎片，拼不出完整的项目认知地图。

### 2. 核心需求（对应你最初的构想）

- **可缩放性（丰俭由君）**：同一套结构，小窗口模型拿顶层摘要就能把握全局，大窗口模型可以逐层下钻到细节，任意层级都能拿到 “该粒度下的完整上下文缩影”。
- **动态自维持**：叶子节点（单个文件、单个函数、单篇文档）发生变更时，只沿目录路径向上逐层刷新摘要，不需要全量重建，整棵树永远是项目的最新状态。
- **分形自相似**：每一层目录的组织逻辑完全一致 —— 都有「功能概述 + 设计意图 + 风格约定 + 关键依赖 + 变更记录」，目录套目录，结构无限递归也不会乱。
- **保留 Vibe 而非只保留功能**：摘要里不止有 “做了什么”，还要有 “为什么这么做”“不能怎么做”“整体风格是什么” 这些隐性上下文，这是 VibeCoding 的核心。

### 3. 主体关联

- **天然骨架**：项目的目录树结构本身就是现成的分形载体，不需要额外设计数据结构，直接在每一层目录挂载一个上下文摘要文件即可。
- **服务对象**：不同上下文长度的大模型、AI 编程助手（Cursor、Claude Dev 这类）、以及项目里的新成员（人也可以用这套结构快速上手）。
- **工程边界**：属于「上下文工程」的核心子集，和 Harness 工程强相关 —— 本质是用工程化手段纠正模型 “只见树木不见森林、容易丢失全局一致性” 的本能缺陷。

### 第 2 层・第二子项：特殊情况（因地制宜 / 一方水土养一方人）

核心是：**结构自相似，但内容权重不千篇一律**，不同场景下分层摘要的侧重点要适配场景属性，不能用一套模板硬套所有项目。

1. **项目类型差异**
    
    纯逻辑类后端项目，每层摘要侧重接口契约、依赖关系、性能约束与异常处理约定；创意类前端 / 设计项目，要加重设计风格、交互调性、视觉一致性、命名审美等软属性的占比；研究类 / 文档型项目，则分层侧重论点脉络、论据层级、结论演化与假设前提。
    
2. **层级深度差异**
    
    越靠近顶层（根目录），越偏向 “为什么”—— 设计哲学、技术选型、全局约束、核心目标；越靠近叶子节点（单文件 / 单函数），越偏向 “是什么”—— 实现逻辑、参数细节、边界条件、踩坑记录；中间层承上启下，核心讲模块间的协作关系、边界划分与对外承诺。结构逻辑自相似，但内容重心逐层渐变。
    
3. **变更频率差异**
    
    高频迭代的业务模块目录，需要更短的刷新周期、更细粒度的变更日志；稳定的基础库 / 工具层目录，可以降低刷新频率，加重稳定性结论与使用规范的权重。
    
4. **使用主体差异**
    
    给小窗口模型供给时，摘要要极致压缩结构，只保留核心骨架与硬约束；给大窗口模型或人类开发者使用时，可以保留更多决策背景、反例与取舍逻辑。
    

### 第 2 层・第三子项：内涵与外延（是什么 / 不是什么）

### 它是什么

- 是一套**分形递归的上下文抽象体系**：以项目原生目录树为天然骨架，每一层都生成对应粒度的 “完整上下文缩影”，组织逻辑自相似，可无限嵌套扩展。
- 是一种**增量式动态上下文维护机制**：底层节点变更仅沿目录路径向上传导刷新，无需全量重建，始终保持全局上下文与项目状态同步。
- 是一种**多窗口适配的上下文供给协议**：对不同上下文长度的模型，输出对应粒度的结构化上下文包，真正实现 “丰俭由君”。
- 是 VibeCoding 场景下的**隐性知识沉淀载体**：不止沉淀 “做了什么” 的功能信息，更沉淀设计意图、风格约定、禁忌边界、历史决策等软上下文。

### 它不是什么

- 不是普通代码摘要工具：普通摘要仅做功能层面的自然语言转述，无全局层级结构，也不支持递归分层与动态增量更新。
- 不是传统 RAG 检索系统：RAG 是 “碎片化召回 + 随机拼接”，没有全局空间结构，模型无法建立完整的项目认知地图；这套体系是 “结构化树状导航 + 按需下钻”，自带全局坐标系。
- 不是静态项目文档：README、架构文档是人工维护的静态快照，会随项目迭代快速过时；这套体系是随代码自动演进的活的认知层。
- 不是版本控制系统：它不管理代码本身的版本，只管理 “认知层面的上下文版本”，是对 Git 等工具的认知层补充。

### 第 2 层・第四子项：与三类工程的定位关联

对应你提到的提示词工程、上下文工程、Harness 工程：

1. **与提示词工程：上游供给关系**
    
    提示词工程解决单次提问的表达精准度，是 “单点输入优化”；这套分形上下文是前置的基础设施，给所有提示词提供结构化、高一致性的上下文基底 —— 优质提示词的效果上限，很大程度由底层上下文质量决定。
    
2. **与上下文工程：核心分支落地**
    
    它本身就是上下文工程的一个核心范式 ——**结构化分层上下文工程**。传统上下文工程更多关注截断、排序、去重、注入时机；这套方案解决的是 “无限规模下的上下文可缩放组织”，是面向超大型、长周期项目的上下文工程升级。
    
3. **与 Harness 工程：结构约束类方案**
    
    Harness 工程的核心是纠正模型的本能缺陷（幻觉、局部最优、全局一致性丢失等）。这套分形体系直接针对模型 “长上下文注意力稀释、缺乏全局结构感、易偏离项目调性” 的固有问题，用分层摘要强制锚定全局约束，属于 Harness 工程里的 “结构矫正类方案”。

### 3.1 历史进展与相关材料脉络

我按时间线拆成三个阶段，你看有没有遗漏的关键节点或相关实践：

### （1）前 AI 时代：软件工程的分层抽象传统（思想源头）

这套分形结构的底层逻辑，本质是人类管理复杂软件系统的成熟方法论的延续，只是把服务对象从 “人” 换成了 “模型”。

- **模块化与信息隐藏原则**：1970 年代 Parnas 提出信息隐藏、Dijkstra 奠定结构化编程，核心思路就是 “拆与封”—— 把大系统拆成独立模块，每个模块对外只暴露必要的接口契约，隐藏内部实现细节。这和我们 “每层目录对外输出摘要、隐藏内部文件细节” 的逻辑完全同源，区别只是过去的接口是给程序调用、给开发者看的，现在的摘要是给模型读取的结构化上下文。
- **组合模式与递归架构**：GoF 设计模式中的 Composite（组合模式）是最典型的工程分形实现 —— 叶子对象和组合对象对外接口一致，可以无限嵌套成树状结构。后续分布式系统里的 “自相似架构”“分形部署” 也是同一思路：每一层节点的运行逻辑完全相同，规模可无限伸缩。
- **人工分层文档体系**：传统软件项目的「详细设计文档→概要设计文档→架构说明书」，本身就是一套人工维护的分层上下文，粒度从细节到全局逐级提升。它的核心痛点是人工维护成本高、静态易过期，我们的方案本质是把这套体系自动化、动态化、全递归化。

### （2）大模型早期：上下文工程的探索与瓶颈

大模型出现后，上下文从 “给人读” 变成 “给模型喂”，旧的线性思路走到了瓶颈，催生了两代解决方案：

- **第一代：全量灌入与线性优化**：早期小项目直接全量塞代码，完全依赖模型窗口大小；后续衍生出按重要性排序、去重、截断等优化，但本质还是线性平铺，没有全局结构，规模稍大就失效。
- **第二代：RAG 检索增强**：当前主流方案，通过向量召回相关代码片段补全上下文。但它的原生缺陷我们之前提过：是碎片化的随机拼接，没有全局结构坐标系，模型拿到的是零散拼图，既拼不出完整的项目认知，也感知不到整体 Vibe 与设计哲学。
- **过渡形态：单层 / 双层项目摘要**：部分工具与开发者开始尝试 “文件级摘要→项目级总览” 的两层结构，但大多止步于两层，没有递归到每一级目录；且是全量静态生成，没有增量更新机制，项目一改就要全量重算，效率极低。

### （3）近期：分形上下文的雏形探索

也就是你提到的 “有人和你想法差不多” 的阶段，目前还停留在零散探索，没有形成成熟范式：

- 开发者社区陆续出现「递归目录摘要」「分形项目上下文」的思路讨论，核心都是复用原生目录树的自相似结构做逐层归总；
- 主流 AI 编程工具（如 Cursor 项目索引、Claude Projects 文件分组）开始出现分层组织的雏形，但尚未做到严格的分形递归、增量更新，也没有明确的 “多粒度自适应供给” 协议；
- 整体还停留在 “解决大项目代码理解” 的实用层面，没有上升为一套通用的上下文工程范式。

### 3.2 相关哲学思想锚定

我逐个对应你列出的哲学家，找思想上的同构性，而不是生硬套用，你可以看看哪些契合度高、哪些比较牵强：

### （1）亚里士多德 / 笛卡尔：分层抽象的逻辑根基

- **亚里士多德**：一是**整体论**—— 整体大于部分之和，项目的 Vibe、架构哲学不是单个文件的简单相加，而是在分层归总中涌现的整体属性；二是**范畴与种属层级**—— 从顶层 “种”（项目整体）到中层 “属”（模块）再到底层 “个体”（函数 / 代码块），每一层都是对下一层的本质归类，和我们的分层层级完全同构。
- **笛卡尔**：他的**分解 - 还原法**—— 把复杂问题拆解到最小可清晰认知的单元，再逐级组合回整体。正好对应我们体系的双向通路：向下拆解是从顶层摘要下钻到叶子细节，向上归总是从叶子单元还原出整体认知，是笛卡尔理性方法论在上下文工程里的落地。

### （2）莱布尼茨 / 德勒兹：分形自相似的思想源头

这组和 “分形递归” 的核心特征契合度最高。

- **莱布尼茨・单子论**：每个单子都是宇宙的一面镜子，以自身的清晰程度表象整个宇宙；单子有从微知觉到统觉的等级序列，越高级的单子对整体的表象越清晰、对细节的感知越收敛。
    
    完美对应我们的节点体系：每一层目录都是一个 “单子”，以自身粒度表象项目的全局关联；叶子节点是低级单子 —— 细节最清晰、全局最模糊；根目录是高级单子 —— 全局最清晰、细节被收敛；逐层归总就是从微知觉上升到统觉的过程。
    
- **德勒兹・褶子论**：德勒兹解读莱布尼茨提出 “褶子” 概念 —— 物质无限折叠，每一个折叠里都包含着整体，展开就是细节，折叠就是抽象。
    
    对应我们的分层结构：顶层是折叠起来的项目整体，每下钻一层就是展开一次褶子，露出更细的细节；每一层的折叠规则完全一致，因此可以无限嵌套。
    

### （3）黑格尔：辩证扬弃与动态整体

对应体系里 “向上归总” 和 “动态更新” 的底层逻辑。

- **扬弃（Aufheben）**：辩证法的核心，上升过程不是简单删减，而是既否定片面、偶然的部分，又保留本质、必然的部分。我们每一层摘要的生成，不是对下一层做字数压缩，而是扬弃 —— 扔掉具体变量名、临时写法等偶然细节，保留功能、设计意图、硬约束等本质内容，上升到更高的抽象层级。
- **整体即真理**：只有在整体体系中，部分的意义才能被准确理解。这正好戳中现有上下文方案的痛点：模型只看局部代码，就会脱离整体语境写出违和内容；而分形体系让局部永远处在整体视域里，保证了认知与风格的一致性。

### （4）胡塞尔现象学：本质还原与视域结构

对应 “摘要生成逻辑” 和 “多粒度适配” 的认知底层逻辑。

- **本质还原（悬置）**：把具体、偶然的经验事实存而不论，直接把握事物的不变本质结构。这就是我们生成每层摘要的核心方法论：悬置具体的代码实现、临时的业务逻辑，还原出模块的本质功能、核心约束、设计意图、风格调性这些稳定的本质结构。
    
- **视域（Horizon）理论**：任何意识都有视域结构，有核心、有边缘，有指向内部细节的内视域和指向外部关联的外视域；不同认知深度对应不同视域范围。
    
    完美解释 “丰俭由君” 的底层逻辑：小窗口模型对应窄视域，只拿顶层核心摘要；大窗口模型对应宽视域，可以下钻更多边缘细节。这也是它本质上优于 RAG 的地方 ——RAG 没有视域，只有碎片。
    

### （5）卢曼：系统论与复杂度消减

对应体系的核心功能和分层运作逻辑。

- **系统降低环境复杂度**：卢曼认为系统的核心作用，是通过边界和规则过滤无限的环境复杂度，维持自身运作。我们这套体系的核心价值正在于此：把无限膨胀的项目代码（环境复杂度），通过分层过滤变成对应粒度的结构化摘要（系统内的有限信息），让有限窗口的模型也能处理无限规模的项目。
- **自创生与分层系统**：每层系统都遵循相同的运作逻辑，自己生产自身结构（自创生），同时通过沟通媒介与上下层互动。对应我们的每一层目录，都遵循完全相同的摘要模板（自相似），自己生成自身的上下文摘要，同时通过摘要与上层、下层系统做信息交互。

### （6）杜威：实用主义与动态经验

对应体系的工具属性和动态特性。

- **工具主义**：思想是解决问题的工具，真理的标准是效用。这套分形上下文本质就是实用工具，没有 “绝对正确的摘要”，只有 “适配当前任务、当前窗口大小的有效摘要”；它不是最终目的，而是服务于 VibeCoding、提升模型编码一致性的脚手架。
- **经验的连续性**：认知是在行动中连续演化的，不是静态的。对应我们的增量更新机制：上下文不是一次性生成的静态文档，而是随着项目代码变更持续演化的连续经验，永远和项目状态同步。

### （7）弗雷格 / 卡尔纳普：语言层级与可证实性

对应体系的正确性约束，用来对抗模型幻觉。

- **语言层级理论**：弗雷格区分对象语言和元语言，元语言描述对象语言，层级不可混淆。我们的摘要体系天然有严格的语言层级：下层的代码 / 文档是对象语言，上层的摘要是元语言；每一层的表述都必须能还原到下一层的具体内容，不能跨层编造。
- **可证实原则**：卡尔纳普认为一个陈述有意义的前提是可以被经验证实。这正好可以作为摘要生成的约束规则：每一层摘要里的每一句话，都必须能在下层具体内容里找到依据，不能生成无中生有的幻觉内容，从结构上保证上下文的真实性。

# Skill: FractalContext 分形项目上下文管理

## 技能标识

- 技能 ID：`fractal-context-manager`
- 适用 Agent：支持文件系统读写的代码类 Agent（Claude Code、Cursor Agent 等）
- 所属领域：上下文工程 / Harness 工程

## 核心设计原则（执行所有操作必须遵守）

1. **分形自相似原则**：每一层目录的上下文文件结构完全一致，既是上层的组成部分，也是下层的整体缩影，递归规则全程统一。
2. **增量传导原则**：叶子节点变更仅沿目录路径向上刷新，禁止无差别全量重建，保证时效性与执行效率。
3. **可溯源原则**：所有上层摘要表述必须能对应到下层具体内容，禁止生成无依据的幻觉性描述。
4. **Vibe 优先原则**：摘要不止沉淀功能信息，更需沉淀设计意图、风格约定、决策背景、禁忌边界等隐性上下文。
5. **无侵入原则**：仅在各级目录新增专属上下文文件，不修改项目原有代码与结构；默认支持纳入 `.gitignore`。

## 核心工作流

### 一、初始化构建流程

**触发场景**：首次在项目中启用本分形体系、项目大规模重构后需重建上下文树

**执行命令**：`/fractal-init`

### 执行步骤

1. **前置扫描与规则加载**
    - 遍历项目根目录，识别项目类型、技术栈与整体规模
    - 加载忽略规则：默认排除 `.git`、`node_modules`、`dist`、`build`、`venv`、`.next` 等构建产物与依赖目录，同时读取项目 `.gitignore` 补充自定义规则
    - 设定分形深度阈值：超大型项目可配置最低生效深度，避免过深的工具类目录过度碎片化
2. **自底向上逐层生成**
    - 从最深层级的目录开始处理，依次向上递归直到根目录
    - 对每个目标目录：读取其下所有业务文件内容 + 所有子目录的分形上下文文件
    - 按照「标准上下文文件模板」生成本目录的 `.fractal-context.md`
    - 生成规则：功能做客观归纳，设计做本质提炼，禁忌做明确标注，禁止冗余复述代码细节
3. **收尾索引**
    - 在根目录生成 `.fractal-index.json`，记录全部分形上下文文件的路径、层级与最后更新时间，作为后续增量维护的索引
    - 自动向项目 `.gitignore` 追加分形上下文相关文件的忽略规则（可手动取消）

**输出产物**：全项目每级目录下的 `.fractal-context.md` + 根目录索引文件

### 二、增量维护流程

**触发场景**：文件 / 目录发生新增、修改、删除后自动执行，或手动指定路径更新

**执行命令**：`/fractal-update [目标路径]`

### 执行步骤

1. **变更定位**：识别本次变更的所有文件 / 目录的完整绝对路径，确定受影响的最底层节点
2. **路径向上传导刷新**
    - 从变更节点的父目录开始，更新 `.fractal-context.md` 中对应文件 / 子目录的条目
    - 逐级向上递归，依次刷新每一级父目录的相关内容与结构概览
    - 直达根目录，完成全链路更新
3. **变更留痕**：在每一级被更新的上下文文件「变更记录」章节，追加本次变更的核心内容与时间戳
4. **边界约束**：仅刷新变更直连路径上的层级，未涉及的目录完全不触碰

### 三、一致性校验与修复流程

**触发场景**：定期巡检、项目结构大改后、怀疑上下文过期或存在幻觉

**执行命令**：`/fractal-check [--fix]`

### 执行步骤

1. **结构完整性校验**
    - 遍历目录树，检查所有非忽略目录是否存在对应的 `.fractal-context.md`
    - 核对每个上下文文件的「下属结构概览」与实际文件 / 目录是否一致，标记遗漏或冗余条目
2. **内容真实性校验**
    - 抽样校验关键目录：上层摘要描述是否与下层实际内容匹配，有无过期、偏离、无依据的幻觉内容
    - 校验规则：任意一句上层结论，必须能在下层文件 / 子上下文中找到明确对应依据
3. **分级修复（加 `--fix` 参数时执行）**
    - 结构缺失：补生成对应目录的上下文文件，并向上传导更新
    - 局部不一致：仅重算对应条目的内容，沿路径向上刷新
    - 大面积偏差：标记受影响范围，建议执行局部目录重建，禁止无差别全量重建

### 四、提炼升华流程

**触发场景**：项目里程碑节点、架构大迭代后、定期（如月维度）执行

**执行命令**：`/fractal-refine`

**核心区别**：不同于日常增量更新的「功能归纳」，本流程聚焦**隐性知识显性化、Vibe 沉淀、架构认知升级**

### 执行步骤

1. **纵向贯通梳理**
    - 从根目录到核心业务模块目录，通读全路径上下文，梳理项目整体的架构演化脉络
    - 提炼贯穿全项目的设计哲学、命名审美、决策偏好、反模式禁忌
2. **隐性知识沉淀**

3. **顶层与中层校准**
